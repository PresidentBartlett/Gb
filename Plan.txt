## TRACK A — AUTOMATED RESEARCH PIPELINE (Google AI Studio, free tier)

### Goal
Replace "paste a prompt into Gemini Deep Research by hand" with a script that takes a procedure name and produces a validated, schema-correct knowledge file with no human in the loop, using Google AI Studio API keys on the free tier.

### Important things to get right before writing code
- **"Deep Research" the consumer product and the plain Gemini API are not the same thing.** The API does not have a one-call "deep research" mode. Check current Google AI Studio / Gemini API documentation directly (don't rely on possibly-outdated assumptions) for:
  - Current available models and their free-tier rate limits (requests per minute, requests per day — these change over time, verify live)
  - Whether a grounding/search-retrieval tool is available via the API (this is the closest equivalent to what made Gemini Deep Research useful — actual live web grounding rather than the model's static training data) and what its current invocation syntax is
  - If no grounding tool is available on the free tier, design the prompt to explicitly ask the model to flag anything it cannot verify with `[FLAG: NEEDS_SOURCE]` rather than silently asserting unverified facts — this matters more without live grounding, not less.
- **Free-tier rate limits are real and will bottleneck a 10+ procedure batch.** Multi-key rotation across keys you legitimately own (separate Google accounts/projects) is reasonable; do not build anything designed to evade per-account quotas through deception (e.g., disposable/fraudulent accounts). Respect documented per-key quotas; rotate and queue, don't circumvent.
- **Never hardcode API keys in source.** Use environment variables or AWS Secrets Manager; the script reads keys from there.

### Build
1. **`research_runner.py`** — CLI tool: `python research_runner.py --procedure passport_application --zone fiscal`
   - Loads the matching prompt set from `Perfected_Haiku_KnowledgePlan.txt` (the 13/14-section template), substituting the procedure name
   - Calls the Gemini API (model and grounding approach as verified above) for each of the 13 sections, or in as few calls as the model's context window allows — decide based on actual output quality you observe in a test run, not assumption
   - Concatenates results into one research document per procedure
2. **Key pool manager (`key_pool.py`)** — Tracks multiple API keys, current usage against each key's known quota, rotates to the next available key on rate-limit (429) response, exponential backoff on transient errors, persists usage counters across runs (so it remembers what's been used today even after a restart)
3. **Converter integration** — Pipes the research document into the existing `proofsheet-converter` Lambda (per `deploy_v2_with_converter.sh`), capturing the returned JSON
4. **Validation step** — Checks the converted JSON against whatever canonical schema Track B has produced by the time this step runs (if Track B isn't done yet, validate against the enum/format findings from Stage 0 as an interim baseline, then re-validate once Track B's schema lands)
5. **Batch orchestration (`batch_regenerate.sh` + `procedures_queue.json`)** — Runs a prioritized queue of procedures, retries failures up to 3 times, logs a manifest (procedure, status, chunk count, flags found, S3 key, timestamp)
6. **Dry run** — Regenerate ONE procedure end-to-end (suggest `passport_application`, highest-volume per the existing roadmap), inspect the output by hand, fix whatever breaks, then run a second procedure to confirm the fix generalizes

### Output
- `research_runner.py`, `key_pool.py`, `batch_regenerate.sh`, `procedures_queue.json`
- `TRACK_A_DRY_RUN_REPORT.md` — what worked, what broke, current verified rate-limit numbers, realistic throughput estimate (procedures/hour at free-tier limits)

## TRACK B — INFORMATION ARCHITECTURE (proceeds now, using the 6 files as sufficient evidence)

### Goal
Finalize the canonical schema and retrieval design without waiting for Track A's regenerated data — the existing six files plus Stage 0's findings are enough evidence to decide on.

### Build
1. **`SCHEMA_CANONICAL.json`** — the full `chunk_type` enum (from Stage 0's derived list, not invented), one-sentence definition per type, the full `intent_tags` vocabulary grouped by category, canonical content-encoding rules (pick `\n` or `<br>` as the one standard — make a decision, don't preserve both), citation handling rule (URLs go in `source_urls` only, never inline in `content`), and migration notes for transforming the six existing files into this schema.
2. **`RETRIEVAL_ARCHITECTURE.md`** — query normalization layer design for Banglish input, OpenSearch lexical+vector retrieval strategy with actual scoring/boosting pseudocode, chunk-selection logic, and the literal system-prompt text for the synthesis step (the words that go into the Lambda, not just a description of the intent).
3. **`DOCUMENT_PIPELINE.md`** — decide HTML-to-S3 (current) vs. JSON-to-React-component (proposed in frontend research) for document generation, with the state-sync mechanism for conversational edits ("change the rent to 15k").
4. **Six corrected sample files** (`NID1_CANONICAL.json`, `TIN1_CANONICAL.json`, etc.) demonstrating the schema actually applied — proof it works, not just a spec.

This track does not depend on Track A finishing. When Track A starts producing newly regenerated procedures, they should validate cleanly against this schema with no further negotiation.

## TRACK C — FRONTEND APP (proceeds now, in parallel, against the currently deployed webhook API)

### Goal
Build the Next.js frontend scaffold against the **existing, already-deployed** webhook Lambda — do not wait for Track A or B. If Track B later changes the response JSON shape, the frontend's API integration layer absorbs that change in one file, not a rewrite.

### Build
1. **Design system** — Tailwind + Shadcn config with the agreed palette (`#1E3F36` primary, `#2A7A71` accent, `#F4F2EE` background, `#C05A46` warning), Inter + Noto Sans Bengali fonts, 1.6 line-height, 44×44px minimum touch targets.
2. **Voice input hook** — Web Speech API wrapper (`bn-BD` locale), real-time interim transcript, graceful fallback to text input when unsupported, lightweight Banglish-to-Bengali normalization pass using the Colloquial.txt lexicon as a starting reference table (not a model call — keep this fast).
3. **Generative UI components** — `ActionCard`, `FinancialRealityCard` (official fee vs. street rate, terracotta for the inflated rate), `TimelineStep` (vertical stepper, color-coded by difficulty), `FlagBadge` (collapsible, color-coded by flag type), enhanced `ChatBubble`, `DocumentPreviewPane`.
4. **State sync hook** — chat history and document JSON as two reconciled state trees, so a message like "change the rent to 15k" patches one field without regenerating the whole document; persisted to localStorage for session resume.
5. **Zero-state dashboard** — logo + language toggle, greeting, auto-scrolling ground-truth ticker, 2×2 expertise card grid (NID / Passport / Tax / Birth Certificate).
6. **Chat interface** — messages pane, voice FAB + text fallback, collapsible document preview pane, wired to the existing webhook endpoint.
7. **API integration layer** — one file (`lib/api.ts`) that talks to the webhook and parses the response into chat text + flags + optional document JSON; this is the seam that absorbs any future schema changes from Track B.
8. **Dry run** — run locally, test voice input with a real Bengali/Banglish phrase, confirm a round trip to the live webhook, confirm a flag renders, confirm the document pane updates on a follow-up edit message.

### Output
- Full `frontend/` Next.js app
- `TRACK_C_DRY_RUN_REPORT.md` with what worked and what's stubbed pending Track B's final document-pipeline decision
